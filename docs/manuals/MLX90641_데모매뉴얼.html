MLX90641 실시간 열화상 데모 — 운용·트러블슈팅 매뉴얼

설치부터 정상 동작 판정, 그리고 초기 구축에서 마주친 두 겹의 소프트웨어 문제 (칩 오인 · ESP32 I2C 드라이버 비호환)의 진단 과정과 증상별 해결까지 한 곳에 정리한 운용 문서.

MLX90641Arduino Nano ESP32 I2C 0x33Web Serial API 115200bps

01개요 · 시스템 구성

MLX90641 적외선 어레이 센서(16×12, 192픽셀)가 읽은 온도 프레임을 Arduino Nano ESP32가 I2C로 읽어 보정 계산한 뒤 USB 시리얼로 PC에 전송하고, 브라우저의 Web Serial HTML 앱이 이를 실시간 열화상으로 렌더링한다. 클라우드도 모델 학습도 없는, 센서 → 보정 → 스트리밍 → 시각화의 순수 측정·시각화 파이프라인이다.

① 센서 측정 ② 보정 계산 ③ 시리얼 스트리밍 ④ 수신·파싱 ⑤ 시각화 16×12 IR 프레임 온도 변환 CSV 한 줄/프레임 Web Serial 캔버스 열화상 + Ta + Vdd (방사율 0.95) 115200bps 깨진 프레임 폐기 통계·트렌드 [MLX90641] ─I2C→ [Nano ESP32] ─USB→ START:..:END ─수신→ [브라우저 앱] ─render→ [화면]
이 매뉴얼을 보는 순서 — 처음 구축이면 02→06을 차례대로 따라가면 동작한다. 값이 이상하면 05(진단)와 11(증상별 진단표)로, 원인 배경이 궁금하면 09(타임라인)·10(라이브러리 분석)을 본다.

03라이브러리 설치

  1. Arduino IDE → 스케치 → 라이브러리 포함하기 → .ZIP 라이브러리 추가…
  2. lib/MLX9064x_Universal.zip 선택 → 설치 완료.
  3. 설치 확인: 파일 → 예제 → MLX9064x_Universal 에 01_Diagnostic, 02_SerialStream_UI, 03_BasicReadings 세 예제가 보이면 정상.
왜 ZIP 한 번이면 끝인가 — 이 ZIP에는 ESP32에서 깨지던 I2C 드라이버를 재작성한 패치본과 칩 자동판별이 이미 들어 있다. 따라서 driver.cpp를 손으로 고칠 필요가 없고, 라이브러리를 재설치해도 패치가 유실되지 않는다. 보정 수학(Melexis 공식)은 원본 그대로라 정확도는 보장된다.

04펌웨어 업로드

  1. firmware/mlx90641_serial.ino 열기 (라이브러리 예제 02_SerialStream_UI와 동일).
  2. 보드 Arduino Nano ESP32 선택, 포트 선택 후 업로드.
  3. 스케치는 래퍼 헤더 MLX9064x_Universal.h 하나만 include 하면 되고, 칩(90641/90640)은 자동 판별되므로 직접 고를 필요가 없다.

핵심 동작만 추리면 다음과 같다:

#include <MLX9064x_Universal.h>
MLX9064xUniversal cam;
float frame[MLX9064X_MAX_PIXELS];   // 어느 칩이든 안전한 크기(768)

void setup() {
  Serial.begin(115200);
  cam.begin();                      // Wire 초기화 + 자동판별 + 보정 로드
  cam.setRefreshRate(MLX9064X_RATE_8HZ);
  // 부팅 1회: INFO:CHIP=MLX90641,COLS=16,ROWS=12
}

void loop() {
  if (cam.readFrame(frame) != 0) return;          // 실패 시 ERR_FRAME
  Serial.print("START:");
  for (uint16_t i=0; i<cam.pixelCount(); i++) {   // 192 또는 768 자동
    Serial.print(frame[i], 2);
    if (i < cam.pixelCount()-1) Serial.print(",");
  }
  Serial.println(":END");
}

05동작 점검 (진단)

스트리밍 펌웨어를 올리기 전(또는 값이 의심될 때) 다음 두 가지로 단계적으로 점검한다.

A. 라이브러리 무관 raw 점검 — firmware/i2c_diagnostic.ino

Wire.h만 사용하는 30초 진단 스케치. 시리얼 모니터(115200)에서:

출력의미
ACK: OK0x33 응답 — 배선·전원 정상
0x240A=0x.... -> MLX90641(16x12)EEPROM bit6 기반 칩 판별 결과
재독취 일치(통신 OK)같은 레지스터 2회 값 일치 — 신호 무결성

B. 라이브러리 예제 — 01_Diagnostic

자동판별·보정까지 포함해 프레임 품질을 본다. 한 줄 출력 예:

Vdd=3.18V Ta= 28.4C | center=  24.9 min=  23.1 max=  25.7 NaN=0

센서에 손을 대면 center·max가 33~36℃로 반응해야 정상이다. 오류 시 readFrame ERR <code> / begin() FAILED, lastError=<code> 형태로 코드를 출력한다(코드표는 08 참조).

06UI 실행 (Web Serial)

  1. IDE 시리얼 모니터를 닫는다. 열려 있으면 포트를 점유해 UI가 연결되지 않는다 (가장 흔한 실패 원인).
  2. ui/mlx90641_thermal_ui.html을 Chrome/Edge에서 연다 (더블클릭, 서버 불필요). 또는 로컬 UI 열기 · GitHub Pages 버전.
  3. 통신 속도 115200 확인 → 장치 연결 → 팝업에서 Nano ESP32 포트 선택.
  4. 센서 앞에 손을 비추면 핫스팟 십자선(H)이 손을 따라가고 최고 온도가 33~36℃를 표시해야 한다.
정상 신호 — 부팅 직후 INFO:CHIP=... 한 줄, 이어서 START:...:END 프레임이 끊김 없이 흐르고, 배경 픽셀이 실온±2℃에서 미세하게 요동치면 정상이다.

09진단 타임라인 — 무엇이 문제였나

초기 구축에서 화면에는 H:7,844℃ 같은 말도 안 되는 핫스팟과 다수의 NaN·Vdd=inf가 떴다. 원인은 하드웨어가 아니라 두 겹의 소프트웨어 문제였고, 다음 순서로 좁혀 나갔다.

  • 증상 포착2026-06

    전 픽셀 NaN·Vdd=inf, 가끔 H:7844℃의 가짜 핫스팟. 배선을 여러 번 점검해도 동일 → 하드웨어 단독 원인 가능성 배제.

  • ① 칩 오인 발견근본원인 1

    모듈을 MLX90640(32×24)으로 가정하고 90640 API를 쓰고 있었으나, EEPROM 0x240A bit6을 직접 읽어 보니 실제 칩은 MLX90641(16×12)이었다. 칩-API 불일치는 보정 추출을 망가뜨려 Vdd=inf·전픽셀 NaN을 만든다. → MLX90641 API로 전환.

  • ② ESP32 I2C 드라이버 비호환근본원인 2

    API를 맞췄는데도 프레임이 동결(매 프레임 동일한 쓰레기 값)됐다. 공식/ Seeed 드라이버의 I2C 읽기가 ESP32 Wire에서 NACK인데도 성공으로 보고하고 requestFrom 수신량을 검증하지 않아, 부분 수신·이전 데이터 잔류가 발생. → I2CRead를 로컬 패치(실패를 -1로 반환 + 수신량 엄격 검증 + 16워드 청크).

  • 재발 방지 + UI 보강대책

    두 수정을 한 라이브러리로 묶고(MLX9064x_Universal) 칩 자동판별을 더해 칩 오인을 원천 차단. UI에는 유효성 게이트(깨진 프레임 폐기)와 컬러맵·보간·시뮬레이터를 구성.

10라이브러리 이슈 분석 (원본 vs 동봉 패치본)

동봉 MLX9064x_Universal은 Seeed/Melexis 원본의 결함만 교체한 포크다. 보정 수학은 손대지 않았다.

항목원본 (Seeed_Arduino_MLX9064x)동봉 패치본
I2C 읽기 실패(NACK)return 0 — 실패를 성공으로 보고-1 반환, 호출부 재시도 가능
requestFrom 수신량미검증 → 부분 수신 시 이전/0xFF 잔류바이트 수 엄격 검증, 미달 시 오류
청크 크기플랫폼 매크로 의존(충돌 가능)16워드(32바이트) 고정 — 전 플랫폼 안전
칩 선택사용자가 90640/90641 직접 선택(틀리면 Vdd=inf)EEPROM 0x240A bit6 자동판별
보정 수학Melexis 공식동일 (무수정) — 정확도 보장
칩 판별 원리 — EEPROM 워드 0x240A의 bit6(deviceSelect): 1 = MLX90641(16×12, 192px), 0 = MLX90640(32×24, 768px). 펌웨어가 런타임에 이 비트를 읽어 올바른 API를 선택하므로, 같은 스케치가 두 모듈에서 수정 없이 동작한다.

11증상별 진단표

증상원인 / 해결
전 픽셀 NaN, Vdd=inf 칩-API 불일치 (동봉 라이브러리는 자동판별로 예방). 다른 MLX 라이브러리 혼용 여부 확인 → MLX9064x_Universal.zip만 사용.
프레임마다 동일한 비정상 값(동결) 공식/구버전 드라이버 사용 중 → lib/MLX9064x_Universal.zip(패치본)으로 재설치.
컴파일 시 ... .h: No such file 스케치는 MLX9064x_Universal.h 하나만 include (내부 API 헤더 직접 포함 불필요). 구버전 Seeed 라이브러리 잔존 시 삭제.
0x33 미검출 (ACK: FAIL / lastError=-10) Grove 케이블을 I2C 포트(A4/A5)에 연결했는지·전원 LED·케이블 단선 확인, 케이블 교체.
간헐적 ERR_FRAME / -20 I2C 클럭 400k→100kHz 하향, 케이블 최단화·노이즈 차단.
UI에서 포트가 안 보임 IDE 시리얼 모니터 닫기, Chrome/Edge인지 확인, 케이블 재연결(데이터용 USB-C인지 확인).
연결은 되는데 화면이 안 변함 통신 속도 115200 확인, 부팅 INFO:/START: 수신 여부 확인, 보간 OFF로 원본 프레임 점검.
손을 대도 온도가 안 오름 방사율 설정 확인(피부 0.97~0.98), 센서 렌즈 가림·거리 확인, 컬러맵 자동 스케일이 배경에 묶였는지 점검.

12보정 · 교육 활용

기본 보정값

  • 방사율(emissivity) 0.95 기본. 피부는 0.97~0.98, 금속면은 별도 보정 필요.
  • TA_SHIFT 8 기본. 정밀 측정 시 기준 온도계로 1점 교정.
  • API: cam.setEmissivity(float) · cam.setTaShift(float) · cam.getTa() / cam.getVdd().
교육 포인트 — UI에서 보간 OFF + 수동 스케일로 두면 손·금속·유리의 온도가 다르게 보이는 이유(방사율 차이)를 흑체복사·적외선·방사율 단원과 직접 연결해 토론할 수 있다. 같은 실제 온도라도 방사율이 낮은 금속은 더 차갑게 측정되는 현상을 직접 확인시키는 활동이 가능하다.
